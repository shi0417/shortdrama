### 步骤 1：实现“素材筛证代理” (Material Sifting Agent)

**目标**：创建一个新的服务 `MaterialSiftingService`，它能够为任意一集，动态地从数据库中“蒸馏”出我们设计的“戏剧证据包”。

**Cursor 指令 1：创建新服务与 DTO**

请在 `apps/api/src/pipeline` 目录下，对 AI 使用以下指令：

```
@manus-agent

1.  在 `apps/api/src/pipeline/dto` 目录下，创建一个新文件 `material-sifting.dto.ts`，并根据我们之前在 `dramatic_evidence_pack_proposal.md` 中设计的 Schema，定义 `DramaticEvidencePack` 接口。

2.  在 `apps/api/src/pipeline` 目录下，创建一个新文件 `material-sifting.service.ts`，并创建 `MaterialSiftingService` 类。

3.  `MaterialSiftingService` 需要注入 `DataSource` 以便查询数据库。

4.  在 `MaterialSiftingService` 中，创建一个核心的 public async 方法 `buildEvidencePack(novelId: number, episodeNumber: number): Promise<DramaticEvidencePack>`。

5.  在 `pipeline.module.ts` 中，将 `MaterialSiftingService` 加入到 providers 和 exports 中，以便其他服务可以注入它。
```

**Cursor 指令 2：实现 `buildEvidencePack` 核心逻辑**

请打开 `material-sifting.service.ts`，对 AI 使用以下指令：

```
@manus-agent

请实现 `buildEvidencePack` 方法的内部逻辑。你需要分步从数据库中查询数据，并组装成 `DramaticEvidencePack` 结构。参考以下 SQL 查询逻辑：

1.  **`global_context`**: 从 `set_core` 表查询 `novelId` 对应的 `rewrite_goal`, `protagonist_name`, `core_constraint`。

2.  **`temporal_context`**: 
    -   `current_story_phase`: `SELECT * FROM set_story_phases WHERE novel_id = ${novelId} AND ${episodeNumber} BETWEEN start_ep AND end_ep;`
    -   `current_power_level`: `SELECT * FROM set_power_ladder WHERE novel_id = ${novelId} AND ${episodeNumber} BETWEEN start_ep AND end_ep;`
    -   `current_timeline_event`: (可选，初期可留空) 逻辑暂定为从 `novel_timelines` 查找与 `current_story_phase` 关联的事件。

3.  **`character_context`**: 
    -   `protagonist_status`: 从 `novel_characters` 获取主角信息。`immediate_goal` 可从 `novel_episodes` 表中本集的 `outline_content` 提炼。
    -   `active_opponents`: `SELECT * FROM set_opponents WHERE novel_id = ${novelId} AND opponent_matrix_id = (SELECT id FROM set_opponent_matrix WHERE novel_id = ${novelId} AND is_active = 1 LIMIT 1);` (这是一个简化的例子，你可以根据 `set_story_phases` 进一步过滤)。

4.  **`plotline_context`**: 
    -   `active_payoff_lines`: `SELECT * FROM set_payoff_lines WHERE novel_id = ${novelId} AND ${episodeNumber} BETWEEN start_ep AND end_ep;`
    -   `required_hook_rhythm`: `SELECT * FROM novel_hook_rhythm WHERE novel_id = ${novelId} AND episode_number = ${episodeNumber};`

5.  **`source_material_context` (关键技术点)**: 
    -   **初期简化版**: `SELECT content FROM drama_source_text WHERE novel_id = ${novelId} LIMIT 1;` 然后截取前 10000 字作为临时方案。
    -   **最终目标 (需要向量数据库支持)**: 将 `novel_episodes` 中本集的 `outline_content` 作为查询向量，在 `drama_source_text` 的分段文本中进行 Top-K 语义搜索。

请将以上查询结果组装成 `DramaticEvidencePack` 对象并返回。
```

**目标**：改造 `EpisodeStoryGenerationService`，引入一个新的内部方法 `runBeatPlanner`，该方法调用 `MaterialSiftingService` 获取证据包，并执行我们设计的 Beat Planner Prompt。

**Cursor 指令 3：改造 `EpisodeStoryGenerationService`**

请打开 `apps/api/src/pipeline/episode-story-generation.service.ts`，对 AI 使用以下指令：

```

1.  在 `EpisodeStoryGenerationService` 的构造函数中，注入我们新创建的 `MaterialSiftingService`。

2.  创建一个新的 `private async runBeatPlanner(novelId: number, episodeNumber: number): Promise<any>` 方法。

3.  在 `runBeatPlanner` 内部：
    a.  调用 `this.materialSiftingService.buildEvidencePack(novelId, episodeNumber)` 来获取“戏剧证据包”。
    b.  构建一个新的 System Prompt，内容是我们之前在 `beat_planner_prompt_design.md` 中设计的 Beat Planner Agent Prompt。
    c.  构建一个 User Message，内容就是获取到的“戏剧证据包”的 JSON 字符串。

4.  修改 `generateDraft` 方法的逻辑。在生成每一集的 `storyText` 之前，先为该集调用 `runBeatPlanner`，并将返回的 `story_beat_json` 对象暂存起来，与该集关联。
```

## P2 阶段之后：展望 P3 (自动重写代理)

完成以上改造后，你的内容生成流水线将达到一个全新的高度。届时，当 QA 代理发现 `storyText` 不符合 `story_beat_json` 的规划时（例如，反转位置不对），我们就可以启动 P3 阶段的设计：

-   **自动重写代理 (Auto-rewrite Agent)**: 它将接收原始的 `story_beat_json`、有问题的 `storyText` 以及 QA 报告，然后输出一个修正版的 `storyText`。

这会让你的系统具备初步的“自愈”能力，进一步提升自动化生产的效率和质量。

这份实施方案为你铺平了通往“智能编剧”的道路。请按照这些步骤和指令进行操作，如果你在任何一步遇到问题，随时可以向我提问。

### 步骤 4（展望）：设计“自动重写代理” (Auto-Rewrite Agent)

**目标**：当 `Writer Agent` 生成的 `storyText` 未能通过 `Rule-based QA` 时（例如，字数不足、未实现某个节拍），自动触发一个修复流程，而不是直接失败。

**设计思路**: 

1.  **触发时机**: 在 `persistDraft` 流程中，当 `runRuleBasedCheck` 返回 `high severity` 错误时。
2.  **输入**: 
    -   原始的 `story_beat_json` (作为“标准答案”)
    -   失败的 `storyText`
    -   QA 模块返回的结构化错误报告 (e.g., `[{'error_code': 'mid_reversal_not_found', 'message': '未在文本40%-70%处找到反转'}]`)
3.  **Prompt 设计**: 
    -   **角色**: “你是一个经验丰富的剧本医生，你的任务是根据QA报告，修复一段未能严格遵循节拍规划的故事文本。”
    -   **指令**: “请仔细阅读原始节拍规划、有问题的文本和QA错误报告。然后，仅针对错误报告中指出的问题，对故事文本进行最小化的、精准的修改，确保修改后的文本100%符合节拍规划。保持原文的风格和其余正确部分不变。”
4.  **输出**: 修复后的 `storyText`。
5.  **流程**: 修复后的文本可以再次进入 `runRuleBasedCheck` 进行校验。如果通过，则继续 `persist` 流程；如果仍然失败（设置一个最大重试次数，如1次），则最终失败并向用户报告。

**这将是 P3 阶段的核心功能，它将为你的内容生产线带来“自愈”能力，是迈向更高阶自动化的关键一步。**
