# Episode Story 短剧可拍性评估标准与修复规范

本文档定义 episode_story_versions.story_text 的短剧可拍性评估标准、生成规范与修复策略，确保产出内容适合 60/90 秒短剧旁白驱动表达，且终局服从项目 rewrite_goal。

---

## 1. 目标定义

### 1.1 story_text 的定位

- **不是**：普通剧情介绍、小说正文、第三人称简介、分镜头脚本、纯梗概。
- **而是**：**短剧连续旁白正文**，在短剧中承担持续旁白介绍剧情的功能。

### 1.2 必须满足的要求

| 维度 | 说明 |
|------|------|
| **第一人称** | 统一为第一人称叙述，默认旁白者为女主沈照 |
| **可口播** | 像人能直接念出来，适合配音 |
| **可承接画面** | 能与画面配合，可拍、可剪 |
| **有节奏** | 单集内有开场抓力、中段推进、结尾钩子 |
| **有单集钩子** | 每集结尾形成下一集必点 |
| **有跨集推进** | 与上一集、下一集因果相连，不脱节 |

---

## 2. 当前内容问题归类

| 类别 | 表现 |
|------|------|
| **叙述视角问题** | 大量“沈照如何如何”“她如何如何”，像第三人称摘要，不像旁白 |
| **爆点不足** | 缺少“当场识破/拦下/翻盘/拿到证据”等可拍爆点 |
| **爽点不够具象** | 反击/揭露/翻盘/掌控不具体，无明确压制对象与可见结果 |
| **钩子模板化** | 结尾多为“风暴将至/暗流涌动/局势紧张”，不具体 |
| **中后段重复** | 重复“局势紧张、暗流涌动、沈照布局”等模板句 |
| **局部事件可拍性弱** | 说明多、画面少，难以直接想到镜头 |
| **主线/阶段漂移** | 中后段偏离主线目标，节奏拖沓 |
| **终局约束失守风险** | 出现朱棣攻破南京、建文朝覆灭等与 rewrite_goal 相反结局 |

---

## 3. Episode Story 短剧可拍性评估标准

每项 0～5 分，5 为完全符合。可做规则或抽样打分。

| 维度 | 0 分 | 5 分 | 说明 |
|------|------|------|------|
| **视角合规** | 全第三人称/无“我” | 全文第一人称旁白（沈照视角） | 是否统一第一人称 |
| **旁白口播感** | 书面/史书总结腔 | 可直接念、有情绪有判断 | 是否像人能念 |
| **单集开场抓力** | 平淡开场 | 前 10 秒有冲突/危险/反常 | 前几句是否抓人 |
| **过程推进度** | 纯说明无动作 | 中段有“我看到/我判断/我准备做”及推进 | 是否有动作推进 |
| **爽点强度** | 无明确反击/翻盘 | 有明确压制对象+主角动作+可见结果 | 是否有爽点 |
| **爆点可视化** | 抽象描述 | 能直接想到画面（识破/拦下/翻盘等） | 是否可拍 |
| **尾钩强度** | 空泛“风暴将至” | 具体到某人/某事/某时（如“那封密折今晚就到”） | 是否必点下一集 |
| **连续性** | 与前后集脱节 | 与上一集、下一集因果相连 | 是否连贯 |
| **阶段主线一致性** | 偏离主线 | 仍在服务主线目标 | 是否不漂移 |
| **终局约束一致性** | 违背 rewrite_goal | 与改写目标一致（建文守江山、朱棣未夺位） | 终局是否跑偏 |

## 4. story_text 生成规范（必须执行）

### 4.1 叙述视角

- **必须统一为第一人称**，默认主叙述者为女主 **沈照**。
- **禁止**大段第三人称“她如何如何”“沈照如何如何”。
- 主叙述句式应以以下为主：
  - “我看到……”
  - “我知道……”
  - “我不能……”
  - “我必须……”
  - “我原以为……”
  - “可我没想到……”
  - “这一刻我才明白……”

### 4.2 文本用途

- **短剧旁白正文**：能直接配音、能压画面、能在短剧里持续讲述剧情推进。
- 不是小说正文、不是第三人称简介、不是分镜头脚本、不是纯梗概。

### 4.3 单集结构建议

每集建议隐含：

- **开头**：快速抛冲突/危机/反常（前 10 秒抓人）。
- **中段**：我看到什么、我判断什么、我准备做什么；有推进。
- **后段**：一次明显推进/反击/发现。
- **结尾**：必须带出下一集风险/倒计时/新变量（具体钩子，非模板句）。

### 4.4 语言风格

- 更像短视频/短剧可口播旁白；句子不要太书面、不要太像史书总结。
- 要有情绪、有判断、有紧迫感；不能油腻、不能过分网文腔。

### 4.5 长度建议

| 版本 | 建议字符区间 | 说明 |
|------|--------------|------|
| 60 秒版 | 约 200～350 字 | 口播约 2～3 字/秒 |
| 90 秒版 | 约 300～500 字 | 口播约 2～3 字/秒 |
| 当前适配 | 建议按 90 秒为主 | 单集 story_text 建议 300～500 字，便于裁剪 |

---

## 5. 爆点 / 爽点 / 钩子标准

### 5.1 爆点

- 必须是能“拍出来”的，不是抽象词。
- 示例：当场识破、当场拦下、当场翻盘、当场拿到证据、当场逼对手露破绽。

### 5.2 爽点

- 必须有：**压制对象** + **主角动作** + **可见结果** + **情绪释放**。

### 5.3 钩子

- 必须具体，不要只写“风暴将至、暗流涌动、危机升级”。
- 要具体到：某个人来了、某封密折到了、某个计划暴露了、某扇门今晚要开、某人下一秒就会背叛。

---

## 6. 终局一致性硬约束

### 6.1 本项目 rewrite_goal

- 沈照改写靖难之役，建文帝守住江山，朱棣不能按历史原线成功夺位。

### 6.2 不合格终局（必须判为不合格）

任何出现以下内容的结果都应判为 **不合格**：

- 朱棣最终攻破南京
- 建文朝覆灭
- 建文帝失败收场
- 主角努力白费但历史未被改写

### 6.3 适用范围

- planner、writer、repair、QA 都必须检查终局是否跑偏；尤其 55～61 集及所有涉及结局的表述。

---

## 7. 修复策略

### 7.1 代码层建议

- 在 episode story 生成完成后增加 **内容 QA 评分**（规则为主，可逐步加维度）。
- 增加 **第一人称旁白校验**：大段第三人称判 invalid 或 warning。
- 增加 **结局一致性校验**：后期集数若出现跑偏关键词，直接判 invalid 并阻止 persist。
- 增加 **模板句重复度校验**：如“局势紧张、暗流涌动、风暴将至”等同批/全局过高时 warning。
- 增加 **钩子弱化校验**：结尾仅为空泛模板句时 warning。
- 若不合格则拒绝 persist 或标记 warnings；终局跑偏必须阻止 persist。

### 7.2 实现优先级

- 规则优先、最小实现；不做复杂 NLP，先做关键词与句式规则。
